# Use Cases / Scenarios

本文件整理第一版預期會出現的實際使用情境，方便後續設計 API、MCP tool、資料模型與 agent 行為。

## 角色

- 使用者：系統唯一的人類使用者
- Agent：透過 MCP tool 與資料系統互動的智能代理
- 未來的 LINE robot：作為互動入口與呈現介面

---

## Scenario 1：新增一隻寵物

### 情境
使用者開始管理新寵物，需要先建立基本資料。

### 使用者可能說法
- 幫我新增一隻貓叫摸摸
- 幫我建立摸摸的資料，母貓，米克斯，生日 2021/03/12，晶片號碼 xxx

### Agent 目標
- 建立 pet 資料
- 回覆已建立成功
- 後續所有紀錄都能關聯到該 pet

---

## Scenario 2：新增病歷

### 情境
使用者剛帶貓去看診，要記錄看診內容。

### 使用者可能說法
- 幫我記錄摸摸今天下午去安心動物醫院看診，醫生是王醫師，診斷是腸胃不適，開 patros 50mg
- 幫我新增摸摸今天的病歷，體重 4.3kg，食慾差，醫生叫我觀察三天

### 系統要支援的內容
- 寵物
- 看診時間
- 醫院名稱
- 醫生名稱
- 診斷
- 處方／用藥
- 備註／病摘
- 體重

### 延伸需求
可再附加：
- 血檢圖片
- 醫生病摘圖片
- 處方箋圖片
- X 光圖片
- 超音波影片

---

## Scenario 3：上傳病歷附件

### 情境
使用者拍了血檢單、X 光或超音波資料，希望附加到病歷中。

### 使用者可能說法
- 這張是摸摸今天的血檢，幫我加到剛剛那筆病歷
- 這是 X 光，請歸類成 xray
- 這段影片是超音波

### 系統要支援的內容
- 檔案上傳
- 附件與病歷綁定
- media_type 區分 image / video
- category 區分 blood_test / xray / ultrasound
- 預留 extracted_text 可手動補入

---

## Scenario 4：新增日常紀錄（文字）

### 情境
使用者在日常生活中簡單記錄貓的狀況。

### 使用者可能說法
- 摸摸今天食慾不振，沒有太多活動力
- 幫我記錄摸摸今天精神不太好
- 摸摸今天有點拉肚子

### 系統要支援的欄位
- logged_at
- content
- appetite
- energy
- stool
- tags

---

## Scenario 5：新增日常用藥紀錄

### 情境
使用者記錄每天用藥時間與劑量。

### 使用者提供範例
- 摸摸早上 7:40 使用 patros 50mg
- 摸摸晚上 8:32 使用 patros 50mg, xxx 100mg

### 系統要支援的內容
- 寵物
- 紀錄時間
- medication_note
- 自由文字內容
- 若有需要，可與附件一起保存

### 後續可延伸
- 用藥摘要
- 每日用藥整理
- 遺漏用藥提醒（第二階段）

---

## Scenario 6：新增日常體重紀錄

### 情境
使用者量到當日體重，要記錄下來。

### 使用者提供範例
- 摸摸 體重 4.3kg

### 系統要支援的內容
- 體重能記在 Daily Log 中
- 日後可彙整成體重趨勢
- 與病歷中的體重一起形成歷史記錄

---

## Scenario 7：新增日常圖片／影片

### 情境
使用者拍下貓咪狀態或錄影，希望記錄成日常觀察。

### 使用者可能說法
- 這張是摸摸今天晚上的狀態
- 這段影片是牠今天呼吸的狀況
- 幫我附到今天的日常紀錄

### 系統要支援的內容
- 圖片／影片上傳
- 與 Daily Log 綁定
- 可補充分類與備註

---

## Scenario 8：依時間查詢病歷

### 情境
使用者想知道某隻貓在某段時間內的所有看診紀錄。

### 使用者可能說法
- 幫我查摸摸最近三個月的病歷
- 找一下去年底到今年初的所有看診紀錄

### 系統要支援的內容
- start / end 時間區間
- 新到舊排序
- 病歷摘要列表

---

## Scenario 9：依關鍵字查詢

### 情境
使用者想查找特定主題，例如血檢、X 光、體重或某藥物。

### 使用者可能說法
- 找出所有血檢資料
- 幫我找 patros 的紀錄
- 查摸摸所有體重紀錄
- 找超音波相關資料

### 系統要支援的內容
- 對病歷與日常紀錄搜尋
- 對附件分類搜尋
- 對用藥與自由文字內容搜尋

---

## Scenario 10：統一健康時間軸

### 情境
使用者不只想看病歷，也想看同時期的日常變化。

### 使用者可能說法
- 幫我整理摸摸最近兩週的健康時間軸
- 我想看上次看診前後的食慾、精神與用藥狀況

### 系統要支援的內容
- 將 Medical Record 與 Daily Log 整合排序
- 依時間顯示完整脈絡
- 讓 agent 能做後續摘要

---

## Scenario 11：修正輸入錯誤

### 情境
使用者發現資料打錯，例如時間、劑量或寵物對象錯誤。

### 使用者可能說法
- 剛剛那筆不是晚上 8:32，是 8:22
- patros 劑量寫錯了，改成 25mg
- 那筆紀錄其實不是摸摸，是另外一隻

### 系統要支援的內容
- 可編輯紀錄
- 能保留 updated_at
- 後續若要做 audit trail 可再擴充

---

## Scenario 12：刪除錯誤紀錄

### 情境
使用者建立錯誤紀錄，想移除，但又擔心誤刪。

### 使用者可能說法
- 幫我刪掉剛剛那筆重複的用藥紀錄
- 那筆病歷打錯了，請作廢

### 系統要支援的內容
- 先顯示刪除摘要
- 確認後才執行
- 使用 soft delete
- 保留 delete_reason

---

## Scenario 13：生成摘要

### 情境
使用者不只是要查資料，還希望 agent 幫忙整理。

### 使用者可能說法
- 幫我整理摸摸最近 7 天的狀況
- 最近一個月體重有變化嗎？
- 看診後這幾天食慾有沒有改善？

### 系統要支援的內容
- timeline query
- 結構化搜尋結果
- 可被 agent 拿去生成摘要

---

## Scenario 14：未來整合 LINE robot

### 情境
未來使用者透過 LINE 傳文字、圖片、影片給 bot，由 bot 背後交給 agent + MCP tool 處理。

### 可能流程
1. 使用者傳送訊息與附件到 LINE
2. LINE robot 收到請求
3. Agent 解析內容
4. Agent 呼叫 MCP tool 寫入／查詢資料
5. LINE robot 回傳整理後的結果或報表

### 第一版對此的要求
- backend / MCP 介面要穩定
- 不依賴特定前端平台
- 回傳資料格式清楚，方便二次呈現

---

## 總結

第一版的核心 use case 可歸納成：
1. 管理多隻寵物
2. 記錄病歷
3. 記錄日常狀態
4. 上傳圖片／影片附件
5. 查詢時間區間與關鍵字
6. 修正或作廢錯誤紀錄
7. 讓 agent 可以基於資料進一步整理與摘要
